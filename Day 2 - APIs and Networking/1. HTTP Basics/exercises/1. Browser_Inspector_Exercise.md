# Browser Inspector Exercise: Exploring HTTP in Action

> **Tools used:** Browser DevTools (Network tab + Console)
> **Playground APIs:** `https://jsonplaceholder.typicode.com` · `https://httpbin.org`

---

## Setup

1. Open Google Chrome (recommended) or Firefox
2. Press `F12` or right-click → **Inspect**
3. Go to the **Network** tab
4. Check **Preserve log** — this keeps all requests visible even after navigation
5. Keep DevTools open for the entire exercise

---

## Exercise 1: Observing GET Requests

Every time your browser loads a page, it fires HTTP requests. Let's watch them happen.

### Task 1: Your First GET Request
1. With the Network tab open, visit `https://jsonplaceholder.typicode.com/posts`
2. Click the first request in the list (the one to `/posts`)
3. Explore the tabs on the right panel:
   - **Headers** — request and response metadata
   - **Preview** — formatted JSON response
   - **Response** — raw response body
   - **Timing** — how long each phase took

Questions to answer:
- What HTTP method was used?
- What is the response status code?
- What `Content-Type` did the server respond with?
- How many milliseconds did the request take? (check the Timing tab)

### Task 2: Dig Into the Headers
Click on the `/posts` request and open the **Headers** tab. Find:

**Request headers** (what your browser sent):
- `User-Agent` — what does your browser identify itself as?
- `Accept` — what formats did your browser say it can handle?

**Response headers** (what the server sent back):
- `Cache-Control` — how long should this response be cached?
- `X-Powered-By` — what server technology is running?
- `X-RateLimit-Limit` — how many requests are you allowed?

### Task 3: Filter and Explore
Use the **filter bar** at the top of the Network tab:
- Type `posts` to show only requests matching that path
- Click the **XHR** filter button — this shows only API/data requests (no images, CSS, fonts)
- Click **All** to go back to seeing everything

## Exercise 2: Making HTTP Requests from the Browser Console

The browser Console lets you fire HTTP requests using the `fetch()` API. Open the **Console** tab (next to Network), paste the code, and watch the request appear live in the **Network** tab.

> **Tip:** Switch between Console and Network tabs as you work — paste in Console, then click over to Network to inspect the result.

### Task 1: GET — Fetch a Single Resource

```javascript
fetch('https://jsonplaceholder.typicode.com/posts/1')
  .then(response => {
    console.log('Status:', response.status);
    console.log('Status Text:', response.statusText);
    return response.json();
  })
  .then(data => console.log('Data:', data));
```

Switch to the Network tab after running this. Click the new `/posts/1` request and answer:
- What status code came back?
- What fields does the post object have?
- Can you see the response body in both the Console and the Network → Preview tab?

### Task 2: GET with Query Parameters — Filter Results

```javascript
fetch('https://jsonplaceholder.typicode.com/posts?userId=1')
  .then(res => res.json())
  .then(data => console.log(`Found ${data.length} posts for userId=1`, data));
```

In the Network tab, click the request and look at the **Headers** tab:
- Find the full request URL — can you see the `?userId=1` query string?
- Try changing `userId=1` to `userId=3` in the Console and compare results

### Task 3: POST — Create a New Resource

```javascript
fetch('https://jsonplaceholder.typicode.com/posts', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    title: 'My First Post',
    body: 'Hello from the browser console!',
    userId: 1
  })
})
  .then(response => {
    console.log('Status:', response.status);
    return response.json();
  })
  .then(data => console.log('Created:', data));
```

In the Network tab, click the POST request and explore:
- **Headers** tab → scroll down to **Request Headers** — can you see `Content-Type: application/json`?
- **Payload** tab (Chrome) or **Request** tab (Firefox) — this shows the body you sent
- What status code did you get? (hint: not `200`)
- What `id` was assigned to the new post?

### Task 4: PUT — Full Replace of a Resource

```javascript
fetch('https://jsonplaceholder.typicode.com/posts/1', {
  method: 'PUT',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    id: 1,
    title: 'Completely Replaced Title',
    body: 'This replaced the entire post',
    userId: 1
  })
})
  .then(res => res.json())
  .then(data => console.log('Updated:', data));
```

Questions to answer:
- In the Network tab **Payload**, what did you send?
- What came back in the response? Compare it to what you sent.
- What is the difference between PUT and PATCH?

### Task 5: PATCH — Partial Update

```javascript
fetch('https://jsonplaceholder.typicode.com/posts/1', {
  method: 'PATCH',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    title: 'Only the title was changed'
  })
})
  .then(res => res.json())
  .then(data => console.log('Patched:', data));
```

Questions to answer:
- You only sent a `title` — what happened to the `body` and `userId` fields in the response?
- Compare this to the PUT response. What is different?

### Task 6: DELETE — Remove a Resource

```javascript
fetch('https://jsonplaceholder.typicode.com/posts/1', {
  method: 'DELETE'
})
  .then(response => {
    console.log('Status:', response.status);
    return response.json();
  })
  .then(data => console.log('Response body:', data));
```

Questions to answer:
- What status code came back?
- What was in the response body?
- In the Network tab, notice there is no **Payload** tab — why not?

## Exercise 3: Inspecting Requests in Depth with httpbin

`httpbin.org` is a purpose-built mirror — it echoes back exactly what your browser sent. This makes it perfect for understanding what headers and data travel with your requests.

### Task 1: What Does the Server See?

Paste this in the Console:
```javascript
fetch('https://httpbin.org/get')
  .then(res => res.json())
  .then(data => console.log(data));
```

Look at the response in the Console:
- What is your public IP address? (the `origin` field)
- What `User-Agent` did your browser send?
- What `Accept` headers did the browser include automatically?

### Task 2: Send Custom Headers and Watch Them Travel

```javascript
fetch('https://httpbin.org/headers', {
  headers: {
    'X-My-Name': 'YourName',
    'X-Course': 'BlitzLearning',
    'X-Exercise': 'HTTPBasics'
  }
})
  .then(res => res.json())
  .then(data => console.log(data));
```

Questions to answer:
- Can you see your custom `X-` headers in the response?
- In the Network tab **Headers** → **Request Headers**, can you also see them there?
- What other headers did the browser add automatically alongside yours?

### Task 3: POST Form Data vs JSON — Spot the Difference

Run these two fetches one after the other and compare the Network tab for each:

```javascript
// Form data
fetch('https://httpbin.org/post', {
  method: 'POST',
  body: new URLSearchParams({ username: 'rens', course: 'blitz' })
})
  .then(res => res.json())
  .then(data => console.log('Form:', data));
```

```javascript
// JSON body
fetch('https://httpbin.org/post', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ username: 'rens', course: 'blitz' })
})
  .then(res => res.json())
  .then(data => console.log('JSON:', data));
```

Questions to answer:
- In the httpbin response, form data appears under `form` and JSON appears under `json` — can you see both?
- In the Network tab **Headers**, what `Content-Type` did each request send?

## Exercise 4: HTTP Status Codes in Action

`httpbin.org` lets you trigger any status code on demand via `/status/{code}`.

### Task 1: Trigger Status Codes and Read Them

Run each of these in the Console and watch the Network tab:

```javascript
// 200 OK
fetch('https://httpbin.org/status/200').then(r => console.log('200:', r.status, r.statusText));

// 201 Created
fetch('https://httpbin.org/status/201').then(r => console.log('201:', r.status, r.statusText));

// 400 Bad Request
fetch('https://httpbin.org/status/400').then(r => console.log('400:', r.status, r.statusText));

// 401 Unauthorized
fetch('https://httpbin.org/status/401').then(r => console.log('401:', r.status, r.statusText));

// 404 Not Found
fetch('https://httpbin.org/status/404').then(r => console.log('404:', r.status, r.statusText));

// 500 Internal Server Error
fetch('https://httpbin.org/status/500').then(r => console.log('500:', r.status, r.statusText));
```

In the Network tab, notice how Chrome/Firefox colour-codes the rows:
- Green/normal — 2xx success
- Orange/yellow — 3xx redirects
- Red — 4xx and 5xx errors

### Task 2: Trigger and Follow a Redirect

```javascript
fetch('https://httpbin.org/redirect/3', { redirect: 'follow' })
  .then(res => {
    console.log('Final URL:', res.url);
    console.log('Final status:', res.status);
  });
```

In the Network tab, look for the chain of requests this created:
- How many requests happened?
- What status code did the redirect responses have?
- What header told the browser where to go next? (check the **Headers** tab of each redirect)

## Challenge Tasks

### Challenge 1: Chain Two Requests in the Console
Fetch a post, extract the `userId`, then use it to fetch that user's full profile — all from the Console:

```javascript
fetch('https://jsonplaceholder.typicode.com/posts/5')
  .then(res => res.json())
  .then(post => {
    console.log('Post belongs to userId:', post.userId);
    return fetch(`https://jsonplaceholder.typicode.com/users/${post.userId}`);
  })
  .then(res => res.json())
  .then(user => console.log('User:', user.name, '|', user.email, '|', user.company.name));
```

In the Network tab, you should see two separate requests fire in sequence. Can you spot them?

### Challenge 2: Catch an Error Response
`fetch()` does NOT throw on 4xx/5xx — you have to check `response.ok` yourself. Try this:

```javascript
fetch('https://jsonplaceholder.typicode.com/posts/99999')
  .then(response => {
    if (!response.ok) {
      throw new Error(`HTTP error! Status: ${response.status}`);
    }
    return response.json();
  })
  .then(data => console.log(data))
  .catch(err => console.error('Caught:', err.message));
```

Questions to answer:
- Did `fetch()` automatically throw because the status was 4xx?
- Why is checking `response.ok` important in real applications?

### Challenge 3: Basic Auth in the Browser
Try authenticating with httpbin's basic auth endpoint:

```javascript
// Correct credentials
fetch('https://httpbin.org/basic-auth/student/password123', {
  headers: {
    'Authorization': 'Basic ' + btoa('student:password123')
  }
})
  .then(res => {
    console.log('Status:', res.status);
    return res.json();
  })
  .then(data => console.log(data));
```

Then try it with wrong credentials and observe the difference in the Network tab.

Questions to answer:
- What does `btoa()` do to the credentials?
- In the Network tab **Headers**, can you see the `Authorization` header you sent?
- What status code came back for wrong credentials?

## DevTools Quick Reference

| Panel | What it shows |
|-------|--------------|
| **Network** | All HTTP requests made by the page |
| **Headers** | Request and response metadata |
| **Payload** | The request body you sent (POST/PUT/PATCH) |
| **Preview** | Formatted response (JSON tree, image, etc.) |
| **Response** | Raw response body text |
| **Timing** | Breakdown of how long each phase took |
| **Console** | Run JavaScript, see logs and errors |

| Network filter | What it does |
|---------------|-------------|
| **All** | Show every request |
| **Fetch/XHR** | Show only API calls |
| **JS / CSS / Img** | Filter by asset type |
| Filter box | Type a keyword to match request URLs |
